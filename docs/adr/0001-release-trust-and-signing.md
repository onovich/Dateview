# ADR 0001: Release Trust, Signing, And Installer Strategy

Date: 2026-06-22
Status: Proposed

## Context

Dateview currently ships as a portable `win-x64` folder/zip preview generated by `scripts\package-release.ps1`.

The P9/P10 release shape is intentionally simple:

- No administrator requirement.
- No HKLM writes.
- No Explorer or taskbar injection.
- No Shell hooks.
- No online holiday API.
- No telemetry.
- No auto-update.
- No installer implementation.

The portable preview includes SHA256 sidecar files and JSON release metadata. SHA256 is useful for integrity checks, but it is not publisher identity and it is not code signing. Windows may still show unknown-publisher, SmartScreen, or Microsoft Defender prompts for an unsigned downloaded executable or zip.

P10 prepares a decision path for release trust. It does not buy certificates, configure signing services, implement an installer, or change runtime behavior.

## Decision Drivers

- Preserve Dateview's normal-user, offline-first, no-injection posture.
- Reduce user confusion and support burden around unsigned Windows warnings.
- Avoid introducing installer complexity before there is a clear distribution need.
- Keep release artifacts reproducible and reviewable.
- Require PM/budget approval before certificate purchase or signing service setup.
- Require a separate implementation phase before installer or auto-update work.

## Options

### Option A: Continue Unsigned Portable Zip

Description:

- Keep the current portable folder/zip distribution.
- Provide SHA256 and release metadata.
- Document unknown-publisher and Windows trust prompts clearly.

Pros:

- Lowest implementation complexity.
- No new dependencies, services, certificates, or installer technology.
- Matches the current no-admin, current-user design.
- Good enough for developer, internal, and small trusted preview distribution.

Cons:

- Windows may show unknown-publisher or SmartScreen warnings.
- Users must trust the distribution source and compare hashes manually.
- Does not provide publisher identity.
- Poor fit for broad public distribution.

### Option B: Code-Sign Portable Executable And Zip

Description:

- Acquire an organization-appropriate code signing certificate.
- Sign the executable and release artifacts in a controlled release pipeline.
- Keep the portable zip as the primary distribution shape.

Pros:

- Improves Windows publisher identity and user trust.
- Keeps the current simple portable distribution.
- Avoids committing to installer technology.
- Can be introduced before or without an installer.

Cons:

- Requires PM/budget/legal ownership of certificate purchase and renewal.
- Requires secure key storage and release-process controls.
- SmartScreen reputation may still need time and real-world distribution history.
- Adds release operations responsibility.

### Option C: MSIX Package

Description:

- Package Dateview as MSIX and sign the package.
- Potentially distribute through a managed channel or direct sideload.

Pros:

- Strong Windows-native packaging and signing model.
- Cleaner install/uninstall lifecycle.
- Better enterprise/manageability story than ad hoc portable zips.

Cons:

- Requires signing and package identity decisions.
- May add user friction for sideloading outside Microsoft Store or enterprise channels.
- Changes the current portable mental model.
- Needs a separate architecture/design phase and implementation validation.

### Option D: Traditional Installer, Such As WiX, Inno Setup, Or NSIS

Description:

- Build an installer that deploys Dateview into a conventional application location and optionally creates shortcuts/startup configuration.

Pros:

- Familiar download/install/uninstall workflow for many users.
- Can reduce manual extraction and path-management support issues.
- Can pair with code signing for better trust UX.

Cons:

- Adds installer technology, scripts, update/migration questions, and support burden.
- May tempt scope creep toward admin installs or machine-wide writes.
- Requires careful design to preserve no-admin and HKCU-only startup behavior.
- Must not be added without a separate ADR/PM decision and validation phase.

## Recommendation

For `0.1.0-preview`, continue with Option A: unsigned portable zip for limited trusted preview distribution, with explicit trust documentation and SHA256 integrity checks.

Before broader public distribution, prefer Option B as the next trust step: sign the portable executable and release artifacts while keeping the portable zip shape. This gives Dateview a clearer publisher identity without immediately introducing installer complexity.

Do not implement MSIX, WiX, Inno Setup, NSIS, or auto-update in P10. Treat those as future distribution decisions after there is evidence that portable zip plus signing is insufficient.

## Required Follow-Up Decisions

- PM/budget decision: whether to purchase and maintain a code signing certificate.
- Ownership decision: who controls certificate private keys and release signing credentials.
- Release-process decision: whether signing happens locally, in CI, or through a dedicated signing service.
- Distribution decision: whether future users need an installer, Microsoft Store/MSIX, enterprise deployment, or portable zip only.
- Support decision: how to communicate SmartScreen reputation and unknown-publisher expectations during preview.

## Consequences

- Dateview remains unsigned for P10.
- SHA256 remains an integrity check, not a signature.
- User docs must continue to be honest about unsigned Windows prompts.
- Any future signing or installer implementation requires a new phase with validation and clear rollback/cleanup behavior.
- Generated release artifacts stay out of git.
